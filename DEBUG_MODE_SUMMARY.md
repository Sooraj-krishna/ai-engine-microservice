# ✅ Code Generation Debug Mode - ENABLED

## What I Did

I've implemented a **debug mode** that saves the raw code generated by Gemini **BEFORE** any validation happens. This is exactly what you asked for - you can now see the code right after generation, without any validation blocking it.

## 🎯 Key Changes

### 1. **Raw Code Logging** (`src/ai_api.py`)

- Added code to save **every single response** from Gemini to: `data/raw_generated_code/`
- Saves BOTH the raw response AND the processed code
- Includes metadata: timestamp, model used, prompt, language
- **Enabled by default** (can disable with `SAVE_RAW_GENERATED_CODE=false`)

### 2. **Validation Results Logging** (`src/validator.py`)

- Saves detailed validation reports to: `data/validation_results/`
- Shows exactly WHY fixes are rejected
- Includes the actual code that was rejected
- Shows approval rates and error patterns
- **Enabled by default** (can disable with `SAVE_VALIDATION_DEBUG=false`)

### 3. **Debug Viewer Script** (`view_debug.sh`)

- Quick command-line tool to view debug files
- Usage: `./view_debug.sh` or `./view_debug.sh raw` or `./view_debug.sh validation`
- No need to manually parse JSON files

### 4. **Documentation** (`DEBUG_CODE_GENERATION.md`)

- Complete guide on how to use debug mode
- Troubleshooting tips
- Examples for common scenarios

## 🚀 How to Use (Quick Start)

### View Latest Generated Code

```bash
# View everything (recommended)
./view_debug.sh

# Or view just raw code
./view_debug.sh raw

# Or view just validation results
./view_debug.sh validation
```

### Manual Check (if you prefer)

```bash
# See latest raw generated code
ls -lt data/raw_generated_code/ | head -5
cat data/raw_generated_code/$(ls -t data/raw_generated_code/ | head -1)

# See latest validation results
ls -lt data/validation_results/ | head -5
cat data/validation_results/$(ls -t data/validation_results/ | head -1)
```

## 📁 Where to Find Files

### Raw Generated Code

**Location**: `data/raw_generated_code/raw_code_<timestamp>.json`

**Contains**:

````json
{
  "timestamp": "2026-01-11T11:50:00",
  "language": "javascript",
  "model_used": "gemini:gemini-1.5-flash-latest",
  "raw_response": "```javascript\n...\n```",  ← EXACT response from Gemini
  "processed_code": "...",                      ← After removing markdown
  "has_markdown": true
}
````

### Validation Results

**Location**: `data/validation_results/validation_<timestamp>.json`

**Contains**:

```json
{
  "total_fixes": 3,
  "safe_fixes": 1,
  "rejected_fixes": 2,
  "validation_results": [
    {
      "fix_number": 1,
      "path": "index.html",
      "is_safe": false,                          ← Was it approved?
      "errors": ["Reason 1", "Reason 2"],        ← Why rejected?
      "content_preview": "...",                  ← The actual code
      "bug_type": "accessibility"
    }
  ]
}
```

## 🔍 Debugging Workflow

### Step 1: Trigger Code Generation

Run your normal workflow that generates code.

### Step 2: Check What Gemini Generated

```bash
./view_debug.sh raw
```

This shows you **exactly** what Gemini returned, before any validation.

**Check for**:

- ✅ Is the code present? → Good!
- ❌ Is `raw_response` empty? → Gemini didn't generate anything
- ❌ Does it contain an error? → API issue

### Step 3: Check Why It Was Rejected (if applicable)

```bash
./view_debug.sh validation
```

This shows you **exactly** why the code was rejected.

**Common rejection reasons**:

- `"Unsafe file location"` → Code tries to modify files outside safe directories
- `"Dangerous pattern"` → Code contains potentially harmful operations
- `"syntax error"` → Code has syntax errors
- `"String balance issues"` → Often false positives

### Step 4: Use the Code Anyway (if you trust it)

If validation rejected the code but you know it's safe:

1. **Option 1**: Copy the code manually from the debug file

   ```bash
   # Extract just the code
   cat data/raw_generated_code/$(ls -t data/raw_generated_code/ | head -1) | jq -r '.processed_code'
   ```

2. **Option 2**: Check the validation results and see the code that was rejected
   ```bash
   cat data/validation_results/$(ls -t data/validation_results/ | head -1) | jq -r '.validation_results[0].content_preview'
   ```

## 📝 Important Notes

### Debug Mode is ENABLED by Default

No configuration needed! Just run your normal workflow and check the `data/` folders.

### To Disable Debug Mode

If you want to disable logging (not recommended):

```bash
export SAVE_RAW_GENERATED_CODE=false
export SAVE_VALIDATION_DEBUG=false
```

### Files are Timestamped

- Format: `raw_code_20260111_115030_123456.json`
- Easy to find latest: `ls -t` sorts by time

### JSON Format

All debug files are JSON for easy parsing with `jq`:

```bash
# Extract just the code
jq -r '.raw_response' data/raw_generated_code/raw_code_*.json

# Find all rejected fixes
jq '.validation_results[] | select(.is_safe == false)' data/validation_results/*.json

# Get rejection reasons
jq -r '.validation_results[] | select(.is_safe == false) | .errors[]' data/validation_results/*.json
```

## 🆘 Troubleshooting

### "No files in data/raw_generated_code/"

- Debug mode might be disabled. Check: `echo $SAVE_RAW_GENERATED_CODE`
- Permissions issue. Try: `chmod 755 data/raw_generated_code`
- Code generation hasn't run yet. Trigger your workflow first.

### "Can't read JSON files"

- Install jq: `sudo apt-get install jq`
- Or use: `cat file.json` to view raw JSON

### "Empty raw_response"

- Gemini API failed to generate code
- Check API key: `echo $GEMINI_API_KEY`
- Check prompt in the debug file - might be malformed
